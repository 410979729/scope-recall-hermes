# Scope Recall Memory Quality Refactor Plan

> **For Hermes:** Use this plan before implementing the memory-quality refactor. Do not start coding until the task list and boundaries are accepted.

**Goal:** Turn `scope-recall` from a raw-turn-heavy recall store into a clean scoped memory substrate that favors curated facts, supports hygiene reporting, avoids `general`/vector pollution, and provides stable write receipts for the governance companion.

**Architecture:** Keep `scope-recall` focused on memory storage, scope isolation, recall, update/merge/delete, read-only hygiene diagnostics, and durable write receipts. Do not move turn audit, candidate decisions, or LLM promotion logic into this plugin. Those belong in `turn-closure-audit`.

**Tech Stack:** Python 3.11+, SQLite FTS, optional LanceDB vector companion, pytest, existing Hermes memory-provider API.

---

## Product boundaries

### In scope

- Stop over-capturing noisy raw turns.
- Filter runtime wrappers before capture.
- Make `general` clearly local scratch, not the main durable memory layer.
- Downrank or optionally exclude `general` from routine recall.
- Prevent `general` from polluting the durable vector companion by default.
- Add read-only hygiene reporting for existing rows.
- Preserve existing scoped truth-store behavior.
- Preserve standalone usability without `turn-closure-audit`.
- Implement the `scope-recall` side of the shared candidate/receipt contract.

### Out of scope

- No new plugin.
- No hard merge with `turn-closure-audit`.
- No blind automatic long-term promotion from raw text.
- No model/provider configuration changes.
- No destructive cleanup by default.
- No schema-breaking migration without explicit migration tests.

## Phase 1 — P0 noise stop

### Task 1.1: Add capture filter module

**Objective:** Centralize skip decisions for runtime wrappers and obvious non-memory text.

**Files:**
- Create: `capture_filters.py`
- Test: `tests/test_capture_filters.py`

**API sketch:**

```python
from __future__ import annotations

import re
from dataclasses import dataclass
from typing import Sequence

@dataclass(frozen=True)
class CaptureFilterResult:
    allowed: bool
    reason: str = ""

DEFAULT_SKIP_PATTERNS: tuple[str, ...] = (
    r"^\[Recent Telegram chat history",
    r"^\[CONTEXT COMPACTION",
    r"Earlier turns were compacted into the summary below",
    r"^Review the conversation above and update the skill library",
)


def should_capture_text(text: str, *, skip_patterns: Sequence[str] = DEFAULT_SKIP_PATTERNS) -> CaptureFilterResult:
    stripped = (text or "").strip()
    if not stripped:
        return CaptureFilterResult(False, "empty")
    for pattern in skip_patterns:
        if re.search(pattern, stripped, flags=re.IGNORECASE | re.MULTILINE):
            return CaptureFilterResult(False, f"skip-pattern:{pattern}")
    return CaptureFilterResult(True, "")
```

**Tests:**

- `[Recent Telegram chat history...]` is rejected.
- `[CONTEXT COMPACTION...]` is rejected.
- ordinary user preference text is allowed.
- empty/whitespace text is rejected.
- runtime-config custom patterns are honored.

**Verification command:**

```bash
pytest tests/test_capture_filters.py -v
```

### Task 1.3: Use filters in `sync_turn()`

**Objective:** Prevent noisy user/assistant content from entering `general`.

**Files:**
- Modify: `provider.py`
- Test: `tests/test_provider.py`

**Implementation approach:**

1. Import `should_capture_text`.
2. After `clean_user` and `clean_assistant`, evaluate filters separately using configured patterns.
3. Allow rule-extracted candidates only from allowed user text.
4. Enqueue raw `turn-user` only when user text is allowed.
5. Enqueue `turn-assistant` only when assistant capture is enabled and assistant text is allowed.
6. Store skip reasons in debug logs only; do not write skip records into memory.

**Regression tests:**

- `sync_turn("[Recent Telegram chat history...]", "ok")` does not create a `general` row.
- `sync_turn("Joy says remember this preference...", "ok")` still allows extracted candidates.
- `capture_assistant=false` prevents assistant rows.
- Structured/list content normalization remains green.

**Verification:**

```bash
pytest tests/test_provider.py -v
pytest -q
```

## Phase 2 — Recall quality and target weighting

### Task 2.1: Make target priority explicit

**Objective:** Ensure curated durable memories rank above raw scratch.

**Files:**
- Modify: `scoring.py`
- Modify: `storage_views.py` if target filtering lives there
- Test: `tests/test_scoring.py`

**Target priority:**

```text
user > memory > project > ops > curated files > general
```

`general` should still be searchable for same-chat/thread continuity, but it must not dominate curated rows.

**Acceptance:** A `user` or `project` row with comparable lexical/vector score ranks above a `general` row.

---

### Task 2.2: Add retrieval config for scratch

**Objective:** Let users tune whether `general` appears in normal recall.

**Config proposal:**

```json
{
  "retrieval": {
    "include_general": "same-scope",
    "general_weight": 0.35
  }
}
```

Allowed modes:

- `same-scope`: default; only current local scope scratch can appear.
- `never`: do not include raw scratch in automatic recall.
- `always`: advanced/debug mode only.

**Acceptance:** Tests prove `include_general=never` suppresses general rows from auto recall while direct search tools can still inspect them.

## Phase 3 — Vector isolation for `general`

### Task 3.1: Exclude `general` from durable vector indexing by default

**Objective:** Prevent raw scratch from contaminating semantic/vector recall.

**Files:**
- Modify: `capture.py` or vector enqueue path if vector writes happen there
- Modify: `vector_store.py` / `vector_runtime.py` as needed
- Test: `tests/test_vector_policy.py` or existing vector tests

**Policy:**

- Durable targets `user`, `memory`, `project`, `ops` may enter the vector companion.
- `general` must not enter the durable vector companion by default.
- If future scratch vector search is needed, use a separate table/namespace or explicit config.

**Acceptance:** A stored `general` row exists in SQLite but does not create a durable vector row when `vector.index_general=false`.
