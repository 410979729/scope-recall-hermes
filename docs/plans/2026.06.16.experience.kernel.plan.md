# Scope Recall Experience Kernel Plan

> **For Hermes:** This is a design/implementation plan. Do not start coding from memory; use this document, then implement task-by-task with tests and live verification.

**Goal:** Upgrade `scope-recall` from a scoped factual/episodic memory provider into a plugin-level **Experience Kernel** that turns successful work into verified, reusable, stale-aware procedural playbooks that can help weaker future models complete similar tasks.

**Architecture:** Keep Hermes core unchanged. Implement the runtime experience loop inside the `scope-recall` plugin using SQLite first: journal/task episodes → playbook extraction → task-time preflight → execution packet → outcome feedback → stale/conflict governance. Skills remain human-readable manuals/anchors/export targets, not the runtime source of truth.

**Tech Stack:** Python 3.11+, Hermes memory-provider plugin hooks, SQLite/WAL/FTS5 truth store, optional vector companion, pytest, existing `scope-recall` doctor/release-gate scripts.

**Date:** 2026.06.16 CST

## 1. Product promise

The target behavior is not “remember more text”. The target behavior is:

```text
User asks for a task
  ↓
Experience preflight classifies the task
  ↓
If a verified reusable playbook exists, return a short execution packet
  ↓
Agent checks live preconditions before applying it
  ↓
Agent follows ordered steps and collects evidence per step
  ↓
Agent verifies final success before claiming done
  ↓
Outcome feedback updates confidence, pitfalls, stale status, or replacement
```

The product promise Joy cares about:

> A weaker future model should become more capable because `scope-recall` gives it a verified procedure, not because the model independently rediscovers the same plan every time.

## 2. Non-negotiable design decisions

### 2.1 Do not modify Hermes core for the MVP

Hermes is the official runtime/body and must remain upgradeable. The feature should ship as a `scope-recall` plugin upgrade.

Use existing plugin surfaces first:

| Existing surface | How Experience Kernel uses it |
|---|---|
| `prefetch(query)` | Inject bounded task-time experience preflight packet. |
| `sync_turn(user, assistant, messages=None)` | Stage task evidence into journal, including optional tool-call evidence when the provider receives `messages`; never write directly into durable playbooks. |
| `on_pre_compress(messages)` if available/runtime-supported | Preserve evidence before compaction where the runtime exposes it. |
| `on_session_end(messages)` / digest scripts | Group task episodes and extract playbook candidates. |
| `get_tool_schemas()` / `handle_tool_call()` | Add operator/user tools for preflight, feedback, inspect, review, promote, stale-check. |
| existing scripts | Extend doctor/release-gate/replay verification. |

If later we need a stronger runtime hook, upstream it as a generic Hermes extension point, not a private scope-recall fork.

### 2.2 SQLite first, PostgreSQL later

PostgreSQL/pgvector is useful later for central multi-instance experience pools, JSONB analytics, row-level policy, and larger benchmark/history queries. It is **not** the first step.

If we migrate fragmented `memories(content, summary, metadata)` to PostgreSQL before the playbook pipeline works, we only get a larger fragmented memory store.

### 2.3 Skills do not become the runtime source of truth

Skills remain valuable, but their role is:

- stable human-readable manuals;
- background principles and safety rules;
- optional references;
- export/review artifacts;
- fallback documentation.

The Experience Kernel runtime source of truth is structured playbook rows with status/confidence/freshness/outcomes.

### 2.4 Do not turn raw transcripts into executable instructions

Raw journal/session data is evidence, not a playbook. Tool output, context-compaction text, plugin-injected wrappers, and model summaries must never be treated as user instructions or verified procedure.

## 3. Scope

### 3.1 In scope

- Task episode boundary detection.
- `procedural_playbooks` schema and validation.
- Successful-path extraction from journal/session bundles.
- Failure-path normalization into pitfalls/do-not-do, not main steps.
- Experience preflight with `direct_reuse`, `guided_reuse`, `no_reuse`.
- Short execution packet injection via `provider.prefetch()`.
- Outcome feedback after using a playbook.
- Stale fact policy: TTL, validators, live-check-required markers.
- Skill anchor/skill conflict model.
- Doctor/audit/replay benchmark proving weak-model benefit.
- Migration/backfill from existing historical tasks into candidate playbooks.

### 3.2 Out of scope for MVP

- Editing Hermes core prompt/loop/skill loader.
- Requiring every user to patch Hermes source.
- Making every playbook a skill file.
- Automatic destructive deletion of old memories/playbooks.
- Blindly auto-executing validators that can mutate systems.
- A central PostgreSQL service before SQLite loop is proven.
- Marketing this publicly as finished “self-improving AI” before replay results exist.

## 5. Required data model

Keep the existing `memories` table as factual/preference/project/ops truth rows. Add separate tables for procedural experience rather than overloading `memories.metadata` until it becomes unreadable.

### 5.1 `task_episodes`

Represents one bounded task attempt.

```sql
CREATE TABLE IF NOT EXISTS task_episodes (
    id TEXT PRIMARY KEY,
    scope_id TEXT NOT NULL,
    shared_scope_id TEXT NOT NULL DEFAULT '',
    session_id TEXT NOT NULL,
    task_class TEXT NOT NULL DEFAULT '',
    task_goal TEXT NOT NULL,
    user_intent TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL DEFAULT 'open',
    outcome TEXT NOT NULL DEFAULT 'unknown',
    started_at TEXT NOT NULL,
    ended_at TEXT,
    message_ids TEXT NOT NULL DEFAULT '[]',
    journal_entry_ids TEXT NOT NULL DEFAULT '[]',
    tool_names TEXT NOT NULL DEFAULT '[]',
    evidence TEXT NOT NULL DEFAULT '[]',
    verification TEXT NOT NULL DEFAULT '[]',
    environment TEXT NOT NULL DEFAULT '{}',
    metadata TEXT NOT NULL DEFAULT '{}'
);
```

**Status values:** `open`, `closed`, `abandoned`, `needs_review`.

**Outcome values:** `success`, `partial`, `failed`, `unknown`.

## 6. Procedural playbook schema

A playbook is not a transcript summary. It is a verified procedure.

```json
{
  "schema_version": "procedural_playbook.v1",
  "task_class": "headscale_one_way_acl",
  "title": "Headscale/Tailscale one-way management ACL",
  "trigger": "User asks to let management machines access a target node while preventing the target from reaching other machines.",
  "goal": "Apply and verify one-way access isolation without losing remote access.",
  "preconditions": [
    {
      "id": "p1",
      "check": "Identify target node and management node(s).",
      "evidence_required": "node names/IPs from live Headscale/Tailscale output"
    },
    {
      "id": "p2",
      "check": "Confirm tested rollback path before ACL apply.",
      "evidence_required": "backup path and command to restore"
    }
  ],
  "steps": [
    {
      "number": 1,
      "capability_class": "read_only",
      "action": "Read current ACL policy and node list.",
      "why": "Avoid editing from assumptions.",
      "evidence_required": "policy file path + live node list",
      "previous_mistakes": [
        "Treating tailscale status visibility as proof of connectivity."
      ]
    },
    {
      "number": 2,
      "capability_class": "local_write",
      "action": "Edit ACL so management source can reach target, but target is not a broad source.",
      "evidence_required": "minimal diff"
    },
    {
      "number": 3,
      "capability_class": "read_only",
      "action": "Run policy validation before reload.",
      "evidence_required": "policy check returns valid"
    },
    {
      "number": 4,
      "capability_class": "service_control",
      "action": "Reload and perform both positive and negative connectivity checks.",
      "evidence_required": "management→target succeeds; target→other blocked/fails as expected"
    }
  ],
  "pitfalls": [
    {
      "signal": "tailscale status still lists nodes",
      "mistake": "Assume listing equals reachable",
      "correction": "Use real TCP/SSH/application connectivity checks."
    }
  ],
  "verification": [
    "policy check passes",
    "service remains active",
    "positive path works",
    "negative path is blocked",
    "rollback backup is retained"
  ],
  "cleanup": [
    "Record backup path and verification output."
  ],
  "reuse_policy": {
    "default_decision": "guided_reuse",
    "direct_reuse_allowed_if": [
      "status == promoted",
      "confidence >= 0.82",
      "preconditions satisfied",
      "not stale",
      "no open skill conflicts"
    ]
  }
}
```

## 7. Experience preflight decision policy

### 7.1 Decision values

| Decision | Meaning | Agent behavior |
|---|---|---|
| `direct_reuse` | Same task class, promoted playbook, fresh enough, no conflicts, high confidence. | Follow playbook steps after live precondition checks; do not reinvent. |
| `guided_reuse` | Similar or partially matching task, stale/uncertain facts, lower confidence, or conflict exists. | Use playbook as scaffold; inspect live state; adapt; verify carefully. |
| `no_reuse` | No reliable playbook or unsafe mismatch. | Normal reasoning/exploration; optionally record episode for future extraction. |

### 7.2 Direct reuse gate

All must pass:

```text
playbook.status == promoted
playbook.confidence >= config.experience.direct_reuse_min_confidence
playbook.task_class matches current task intent
preconditions are checkable
required facts are not stale OR live-check has just passed
no unresolved skill_conflicts for this playbook
scope is accessible to current user/agent identity
playbook is not superseded/quarantined
```

### 7.3 Guided reuse triggers

Any of these should downgrade from direct reuse to guided reuse:

- `confidence` below direct threshold but above useful threshold.
- Last verification expired.
- Environment/host/service version differs.
- Related skill or memory says something materially different.
- Preconditions cannot be fully checked automatically.
- User asks for a variant of a known task.
- A previous `experience_run` recently failed.

### 7.4 No reuse triggers

- No playbook match.
- Match depends on stale mutable fact but no safe validator exists.
- Playbook status is `candidate`, `needs_review`, `superseded`, or `quarantined`.
- Safety/secrets/scope isolation risk.
- Current user instruction conflicts with playbook.

## 8. Execution packet format

`prefetch()` must inject a short bounded packet, not a long manual.

Example:

```text
# Scope Recall Experience Preflight
Decision: guided_reuse
Confidence: 0.76
Task class: headscale_one_way_acl
Playbook: pb_...
Status: promoted
Freshness: needs_live_check for target node and policy file

Use this as a scaffold, not a blind script.

Preconditions to verify first:
1. Identify target node and management node(s). Evidence required: live node list.
2. Confirm rollback path. Evidence required: backup path and restore command.

Ordered steps:
1. [read_only] Read current policy and node list. Evidence: file path + node list.
2. [local_write] Make minimal ACL diff. Evidence: diff.
3. [read_only] Run policy check. Evidence: valid output.
4. [service_control] Reload and verify both directions. Evidence: positive and negative connectivity checks.

Pitfalls:
- Tailscale status visibility is not proof of reachability; use real TCP/SSH checks.

Completion rule:
Do not claim success until every verification item has evidence.
```

Size limits should be configurable and conservative:

```json
"experience": {
  "enabled": true,
  "prefetch_enabled": true,
  "max_packet_chars": 1800,
  "max_steps": 8,
  "direct_reuse_min_confidence": 0.82,
  "guided_reuse_min_confidence": 0.55,
  "require_live_check_on_stale": true
}
```

## 9. Runtime flow

### 9.1 Task start

```text
User message
  ↓
provider.on_turn_start()
  ↓
provider.prefetch(query)
  ↓
Experience classifier detects if query is task-like and non-trivial
  ↓
Search procedural_playbooks by task_class/trigger/entities/FTS5 and optional vector companion
  ↓
Apply freshness + scope + conflict + confidence gates
  ↓
Render short execution packet or return empty/no-reuse note
```

Important: preflight should be quiet for simple Q&A or very short messages. We do not want every chat turn polluted with playbook content.

Search/index contract:

- SQLite truth is authoritative. FTS5 and vector rows are rebuildable indexes, never the source of truth.
- Phase 2 creates a lexical FTS/search path for `task_class`, `trigger`, `title`, `goal`, `preconditions`, `steps`, `pitfalls`, and `verification`.
- Vector companion sync is optional and asynchronous; vector failure must degrade to SQLite/FTS search and be reported by doctor.
- `scope_recall_playbook_search` must expose whether each hit came from exact class match, FTS, vector, or merged RRF ranking.
- Search results must always be filtered by accessible scope IDs before ranking is returned.

### 9.2 During execution

The model receives a short packet. It still must perform tool checks:

- verify preconditions;
- collect step evidence;
- stop/downgrade when preconditions fail;
- use live evidence over old memory;
- verify final state.

### 9.3 Task end / background digest

```text
sync_turn(user, assistant, messages=None) writes journal entries and optional tool-evidence summaries
  ↓
background/session digest groups entries into task episodes
  ↓
if outcome has evidence: extract candidate playbook or playbook update
  ↓
validator checks schema, evidence anchors, redaction, failure normalization
  ↓
candidate stored as candidate/reviewed, not auto-promoted unless policy allows
```

Tool-evidence capture strategy:

- Hermes `MemoryProvider.sync_turn` already supports an optional `messages` keyword, and `MemoryManager.sync_all()` passes it to providers that accept it.
- `scope-recall.provider.sync_turn` must be updated in Phase 4 to accept `messages: Optional[List[Dict[str, Any]]] = None` without changing ordinary capture behavior.
- Tool-call evidence is staged into journal/task episode evidence only after filtering: role must be `tool` or assistant tool-call metadata, content must be redacted, oversized outputs must be summarized, and prompt-injection wrappers must be rejected as instructions.
- Evidence records should store compact provenance (`message_id`, tool name, status, path/command summary, verification snippet), not raw full tool output unless explicitly needed and safe.
- If `messages` is absent, extraction still works from existing journal/user/assistant text but marks tool evidence as incomplete.

### 9.4 After playbook use

```text
experience_run outcome recorded
  ↓
reflection_event created when useful
  ↓
confidence updated
  ↓
pitfalls appended if failure/correction
  ↓
stale/conflict status updated if live evidence contradicted old assumptions
```

## 10. Reflection engine

Reflection must be structured. It should not be a vague assistant self-summary.

### 10.1 Trigger conditions

Run reflection only when at least one is true:

- multi-step task completed;
- tool failures were corrected;
- user corrected the assistant;
- a playbook was used;
- stale fact discovered;
- skill/playbook conflict discovered;
- rollback/recovery occurred;
- task consumed many tool calls or crossed context compression boundary.

### 10.2 Reflection output

```json
{
  "task_goal": "...",
  "outcome": "success|partial|failed",
  "evidence": ["tool outputs or file paths proving the result"],
  "mistakes": ["what went wrong"],
  "root_causes": ["why it happened"],
  "corrections": ["what to do differently next time"],
  "playbook_update": {
    "action": "create_new|append_pitfall|promote|lower_confidence|mark_stale|supersede|no_change",
    "reason": "..."
  }
}
```

### 10.3 Confidence update policy

Start conservative:

| Event | Change |
|---|---|
| Candidate extracted from one successful episode | `status=candidate`, `confidence≈0.55-0.65` |
| Human review approves | `status=reviewed`, confidence +0.10 |
| Replay succeeds | confidence +0.10 |
| Real reuse succeeds with verification | success_count +1, confidence +0.03 to +0.08 |
| Real reuse partially helps | confidence +0.00 to +0.03; add notes |
| Reuse fails due to stale/env mismatch | failure_count +1, stale_count +1, confidence -0.10; maybe `needs_review` |
| User says playbook is wrong/dangerous | immediate `needs_review` or `quarantined` |

Never let confidence increase without evidence.

## 11. Staleness and live-check policy

### 11.1 Why this is mandatory

A correct old fact can become dangerous later: service names, ports, hosts, repo state, API behavior, config paths, release versions, tailnet topology, credentials, model/provider settings.

### 11.2 Truth types

| Truth type | Example | Policy |
|---|---|---|
| `immutable` | A design rationale that does not change | no TTL by default |
| `user_preference` | Joy prefers docs/dirs with dots not underscores | no automatic expiry; user correction supersedes |
| `mutable_local` | service name, file path, port, installed version | TTL + command/file validator if important |
| `mutable_external` | GitHub issue status, package latest version, API docs | TTL + web/API validator |
| `procedure` | ordered steps for task class | expires through outcome feedback and environment constraints |

### 11.3 Validator rules

Validators must be safe by default:

- read-only commands only unless explicitly marked and manually invoked;
- no secret printing;
- timeout bounded;
- output redacted before storage;
- failures mark `needs_live_check`, not automatic false facts;
- destructive validators are forbidden in automatic sweeps.

### 11.4 Runtime behavior on stale facts

If a playbook depends on stale mutable facts, the packet must say:

```text
Freshness: needs_live_check
Do not use the stored fact as true until checked.
Suggested check: ...
```

Do not inject stale facts as plain authoritative context.

## 12. Skill contract

### 12.1 Knowledge priority

At runtime, priority should be:

```text
1. Current user instruction and safety boundaries
2. Live evidence from tools/current system
3. Fresh promoted playbook execution packet
4. Skill background/reference material
5. Ordinary durable memory
6. Model guess
```

### 12.2 Role split

| System | Role |
|---|---|
| Playbook | executable runtime procedure with status/confidence/freshness/outcomes |
| Skill | human-readable stable manual and optional reference |
| Memory row | compact fact/preference/project/ops truth anchor |
| Journal | raw evidence and audit trail |

### 12.3 Conflict behavior

If skill and playbook conflict:

```text
direct_reuse is forbidden
preflight returns guided_reuse
packet names the conflict briefly
agent must live-check before proceeding
reflection records resolution
stale/obsolete source is marked for review
```

Do not silently merge contradictory procedures.

### 12.4 Export behavior

Later, promoted playbooks may be exported to skills or skill references, but this is a one-way human-readable artifact generation step. Runtime should still query structured playbook rows because skills lack confidence/outcome/staleness mechanics.

## 13. Safety and governance requirements

### 13.1 Prompt injection and wrapper hygiene

The extractor must reject or neutralize:

- `[CONTEXT COMPACTION ...]` as active task source;
- `[SYSTEM STATUS ...]` as user instruction;
- plugin-generated advisory text as user fact;
- tool outputs that say “ignore previous instructions”;
- webpages/files that contain instructions to the agent;
- raw logs with accidental secrets.

A generated playbook must point to evidence anchors, but evidence text must not become executable instruction automatically.

### 13.2 Secret handling

- Never store plaintext secrets in playbooks, memories, journal-derived durable rows, vector text, or exported docs.
- Reuse existing redaction logic and add tests with `api_key=`, bearer tokens, GitHub tokens, private-key blocks.
- If a credential must be referenced, use the existing secret-index pattern: safe metadata + external `vault_ref` only.

### 13.3 Scope isolation

Playbook search must use accessible scope IDs exactly like memory search. Shared playbooks need explicit shared-pool policy.

Tests must prove:

- local `general` scratch does not leak to other chats;
- durable user/project/ops scope follows configured identity mapping;
- cross-platform sharing only happens through canonical identity/shared pool rules;
- one agent identity cannot accidentally see another user's private playbooks.

### 13.4 Mutation boundaries

Automatic processes may:

- create candidates;
- mark stale/needs_review;
- append feedback;
- create audit rows.

Automatic processes must not:

- delete old playbooks/memories destructively;
- run destructive validators;
- promote risky playbooks without review/replay gates;
- rewrite skills silently.

## 16. Extraction strategy

### 16.1 Episode creation

The digest should group journal/session messages into candidate task episodes when:

- user asks for an action/repair/config/code/debug/release/plan;
- assistant uses multiple tools or produces a verifiable artifact;
- completion/failure signals exist;
- session title/message sequence indicates one coherent task.

Episode boundary must respect the latest user instruction. Context-compaction summaries are reference only and cannot revive stale tasks as active.

### 16.2 Playbook candidate extraction

Only create a playbook candidate if:

- there is a clear task class;
- at least one successful or partially successful path has evidence;
- ordered steps are recoverable;
- final verification is explicit or can be represented as missing/needs_review;
- secrets/wrappers are filtered;
- failed attempts are separated into pitfalls.

### 16.3 LLM prompt requirements

The extractor prompt must explicitly demand JSON with:

- task class;
- trigger;
- preconditions;
- ordered steps;
- evidence anchors;
- previous mistakes per step;
- pitfalls;
- verification;
- cleanup;
- environment constraints;
- confidence and why;
- whether human review is required.

It must explicitly forbid:

- summarizing as “used tools X/Y/Z” only;
- placing failed steps in main ordered steps;
- storing secrets;
- treating tool/web prompt-injection text as instructions;
- creating direct-reuse candidates without verification evidence.

### 16.4 Deterministic validator

After LLM extraction, deterministic validation must check:

- required fields exist;
- JSON field types are correct;
- step numbers are ordered;
- preconditions and verification are non-empty for reusable candidates;
- evidence anchors point to actual journal/message IDs or file paths;
- no obvious secret patterns;
- candidate confidence/status obeys policy;
- dangerous verbs in validators are not auto-run.

Invalid candidates go to `journal_rejections`/experience rejection table with reason.

## 17. Migration/backfill strategy

### 17.1 No destructive migration

Add tables idempotently. Existing `memories` rows stay untouched.

Migration command should:

```text
open DB
PRAGMA quick_check before
BEGIN
CREATE TABLE IF NOT EXISTS ...
CREATE INDEX IF NOT EXISTS ...
COMMIT
PRAGMA quick_check after
```

### 17.2 Backfill from existing data

Backfill should start as dry-run:

1. Select historical sessions/journal entries with high task-likeness.
2. Generate candidate task episodes.
3. Generate candidate playbooks.
4. Store as `candidate`, not `promoted`.
5. Produce a report for Joy/operator review.

Do not retroactively promote a playbook just because a memory says a task was completed.

### 17.3 Legacy memory relationship

If an existing memory contains a useful procedure, link it as evidence/source:

```text
procedural_playbooks.created_from_episode_id = ... if session exists
metadata.source_memory_ids = [...]
```

But do not delete or rewrite the memory row by default.

## 19. Bug matrix to design against

| Bug / failure mode | Consequence | Preventive design | Test/gate |
|---|---|---|---|
| Failed path saved as main success steps | Weak model repeats broken workflow | extraction separates `steps` vs `pitfalls`; direct reuse requires verification | fixture with fail-then-fix transcript |
| LLM hallucinates missing steps | Plausible but unexecutable playbook | deterministic schema/evidence validator | reject missing evidence anchors |
| Tool-chain summary stored as playbook | Still not useful for weak models | require ordered steps, preconditions, verification | schema test |
| Similar task false positive | Wrong playbook applied | task_class + precondition + confidence gates | preflight mismatch tests |
| Old port/service/path injected as truth | Breaks operations | TTL + `needs_live_check` packet | freshness tests |
| Skill conflicts with playbook | Model mixes old/new instructions | conflict table downgrades direct reuse | skill conflict test |
| Skill + playbook bloats prompt | Token waste and confusion | packet char cap; skill optional anchors | packet size test |
| Weak model ignores packet | No improvement | packet has completion rule; replay benchmark | replay metrics |
| Playbook confidence ratchets upward incorrectly | Bad experience gets stronger | confidence only changes from evidenced outcomes | feedback tests |
| User correction not captured | Repeated mistakes | correction triggers reflection/update | reflection test |
| Context compaction summary treated as current task | Resumes stale work | wrapper filters + latest-user-intent rule | prompt hygiene tests |
| Plugin advisory treated as user fact | SQL/memory pollution | source classification; wrappers rejected | hygiene tests |
| Prompt injection in tool/web/file output | Malicious playbook content | evidence is data; sanitize/filter | injection fixture |
| Secret stored in playbook/vector | credential leak | redaction and secret pattern tests | secret fixtures |
| SQLite migration damages DB | data loss | backup, idempotent schema, quick_check | migration test |
| DB write lock under background digest | lost writes/errors | WAL, short transactions, retries | concurrent test |
| Vector failure disables retrieval silently | quality drop | doctor reports vector/truth mismatch; truth DB authoritative | doctor test |
| Cross-scope leakage | privacy/security incident | accessible_scope_ids on every query | scope isolation test |
| Candidate auto-promoted too early | unsafe direct reuse | candidate/reviewed/promoted state machine | promotion test |
| Stale validator mutates systems | dangerous side effects | read-only validator policy | validator allowlist test |
| Batch freshness sweep too aggressive | needless network/ops load | only high-value rows with validators | sweep dry-run test |
| Public docs overpromise | trust damage | docs gate against implemented features | release review |
| PostgreSQL introduced too early | complexity without product value | SQLite MVP first | architecture review |
| Hermes core patch required | users cannot upgrade cleanly | plugin-only MVP | packaging/install test |
| Existing memory behavior regresses | current users break | all existing tests stay green | full pytest |
| Config JSON regex double-escaping | filters break on save/load | normalize regex; round-trip tests | existing + new config tests |
| Local profile paths hard-coded | wrong Hermes profile affected | use hermes_home/plugin config paths | multi-home test |
| Replay benchmark uses fabricated success | false confidence | fixture expected outputs and explicit scoring | benchmark validation |

## 24. Open design questions to decide before code beyond Phase 3

1. **Auto-promotion threshold:** Should any playbook ever auto-promote from replay + repeated success, or should Joy/operator review be required for all promoted playbooks at first?
2. **Default prefetch:** Should experience preflight be enabled by default once shipped, or installed disabled until user opts in?
3. **Post-MVP shared pool expansion:** MVP default is no cross-user/cross-agent sharing; later we still need to decide which playbooks may be shared across agents/instances and what identity mapping governs them.
4. **Validator execution:** Should validators initially be descriptor-only, with the agent performing the actual tool check, or should a safe read-only validator runner exist in the plugin?
5. **Skill export:** Do we export only manually promoted playbooks to skills, or also reviewed candidates?
6. **Benchmark model:** Which weaker model should be used for replay proof: a cheap local/remote model, Codex lower effort, or a deterministic simulated scorer first?
7. **Retention:** How long should raw task episodes/evidence stay if they are superseded by stable playbooks?

Recommended defaults for safety:

```text
manual promotion first
prefetch opt-in until local dogfood passes
shared pool disabled by default
validators descriptor-only first
skill export manual only
replay starts deterministic/fixture-based, then weak-model live tests
no destructive retention cleanup in MVP
```

## 25. Peer-review hardening amendments

These amendments were added after peer review from sibling agents. They should be treated as plan refinements, not optional commentary.

### 25.1 Pull proof of value earlier

Risk: the original phase order proves the core value too late. If replay waits until Phase 9, we could spend too long building infrastructure before discovering the playbook packet does not materially help weaker models.

Adjustment:

```text
Phase 2.5 — Minimal value proof
  1. Manually create 1-2 playbooks from known historical successful tasks.
  2. Run `scope_recall_experience_preflight` against similar task prompts.
  3. Render the exact packet that would be injected.
  4. Run a dry-run replay or weak-model simulation comparing with/without packet.
  5. Continue heavy extraction work only if the packet improves step quality, verification quality, or tool-call count.
```

Acceptance before Phase 4:

- at least one real historical task class produces a usable packet;
- packet stays under the configured char budget;
- evaluator can see fewer known mistakes or clearer verification behavior;
- if no improvement appears, redesign packet/preflight before building extraction pipeline.

### 25.2 Manual review is mandatory for MVP promotion

Risk: LLM extraction is the single highest-variance part of the design. Schema validation can prove JSON shape and evidence references exist; it cannot prove the extracted procedure is semantically correct.

MVP rule:

```text
candidate -> reviewed -> promoted requires operator review.
No auto-promotion in MVP.
```

Allowed automation:

- create candidates;
- reject malformed/unsafe candidates;
- suggest merges;
- suggest confidence;
- run replay checks.

Forbidden automation in MVP:

- promoting a new playbook to direct-reuse status without human/operator review;
- turning a single successful transcript into `promoted` only because it has evidence anchors.

### 25.3 Feedback closure starts manual

Risk: automatic task-end detection is hard in group chat and long-running agent work. Users may continue a task minutes or hours later; context compression can also obscure the boundary.

MVP rule:

```text
Do not depend on automatic run closure for correctness.
Start with explicit/manual feedback recording.
```

Implementation target:

- `scope_recall_playbook_feedback` records `success|partial|failed|stale|misleading`.
- The agent should call it at the end of a task when it used a playbook and has final evidence.
- Automatic closure can be added later as a helper, but missing automatic closure must not break the feedback loop.

### 25.4 Define the sharing contract in Phase 1

Risk: cross-agent/cross-instance sharing affects scope IDs, portability, skill conflicts, and privacy. Leaving it fuzzy can force schema rework later.

MVP sharing contract:

```text
Default: no cross-user or cross-agent sharing.
Playbooks are scoped like durable memories and filtered by accessible_scope_ids.
Shared pools are disabled unless explicitly configured.
Schema keeps shared_scope_id fields, but direct reuse from shared pools is disabled until policy tests exist.
```

Phase 1 tests must include:

- same-scope playbook visible;
- unrelated scope playbook invisible;
- shared-pool playbook invisible when shared pool disabled;
- shared-pool playbook visible only when explicit config enables it.

### 25.5 Confidence must have a formula, not vibes

Risk: thresholds like `0.82` and `0.55` become mystical if there is no scoring formula.

MVP confidence formula should be simple and inspectable:

```text
base = extractor_confidence or 0.55
review_bonus = 0.10 if reviewed_by_operator else 0.0
replay_bonus = 0.10 if replay_passed else 0.0
outcome_score = (success_count + 1) / (success_count + failure_count + 2)
stale_penalty = min(0.20, 0.05 * stale_count)
recent_failure_penalty = 0.15 if last_run_failed else 0.0
confidence = clamp(0.0, 0.95,
    0.45 * base +
    0.25 * outcome_score +
    review_bonus +
    replay_bonus -
    stale_penalty -
    recent_failure_penalty
)
```

This formula can change after data exists, but the first implementation must be deterministic and explainable.

### 25.6 Track step progress for compaction recovery

Risk: if context compression happens mid-playbook, the model may lose which step it is executing.

Packet/run adjustment:

```text
Experience packet includes:
- playbook_id
- experience_run_id if available
- total_steps
- current_step
- completed_steps summary
- next_required_evidence
```

Feedback/update adjustment:

- `experience_runs.steps_completed` records completed step numbers and evidence.
- If compaction occurs, the next preflight can recover progress from the active/incomplete `experience_run`.
- If progress cannot be recovered safely, downgrade to `guided_reuse` and require re-checking preconditions.

### 25.7 Capability and permission class per playbook step

Risk: a weak model may follow a playbook step that edits files, restarts services, changes ACLs, or touches another instance without noticing the permission boundary.

Each step must carry a conservative capability class:

```text
read_only
local_write
service_control
network_or_remote
cross_instance
credential_adjacent
destructive_or_irreversible
```

MVP behavior:

- `read_only` and safe `local_write` may be included normally.
- `service_control`, `network_or_remote`, and `cross_instance` steps must include explicit preconditions and rollback/verification.
- `credential_adjacent` steps must reference vault/secret-index only, never plaintext.
- `destructive_or_irreversible` steps cannot be direct-reuse in MVP.

### 25.8 Reflection must be gated by value

Risk: full reflection on every task increases latency/token cost and creates more memory noise.

Reflection trigger should be value-gated:

```text
full reflection if:
  playbook used
  OR user corrected the agent
  OR task had failure then recovery
  OR task touched service/config/release/security boundary
  OR tool calls >= configured threshold
  OR final artifact was created/modified
otherwise:
  lightweight/no reflection
```

### 25.9 Sensitive-field minimization belongs in extraction, not only storage

Risk: if sensitive data reaches a playbook candidate, later redaction may miss variants or leave unsafe context in audit rows.

Extraction-phase rule:

- strip or generalize private paths unless needed for reusable procedure;
- store host/service classes rather than full identifiers where possible;
- never include raw API responses if a smaller evidence summary is enough;
- apply redaction before candidate validation, before vector text, and before docs/export.

## 26. Final target state

Today:

```text
scope-recall remembers facts and snippets
agent still often has to re-reason from scratch
weak models do not reliably know the successful path
old facts and skills can still confuse the workflow
```

Target:

```text
scope-recall remembers verified procedures
agent gets a short task-specific execution packet before acting
weak models can follow ordered steps with evidence gates
old facts trigger live-check instead of silent misuse
skills are references, not conflicting runtime authorities
outcomes feed back into confidence and pitfalls
replay benchmarks prove the improvement
```

The core principle for every implementation step:

> **Do not optimize for storing more memory. Optimize for compiling verified successful work into reusable, checkable, stale-aware procedures.**
